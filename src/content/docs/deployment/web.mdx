Pinchana Web is hosted with the repository's `docker-compose.yml`. Compose uses the multi-stage `Dockerfile` to install production dependencies, synchronize the local FFmpeg WebAssembly assets, compile Next.js, and create a non-root runtime image. The default local tag is `pinchana-web:latest`; override it with `PINCHANA_WEB_IMAGE` when the host needs a different tag.

Pinchana Web requires a running server. It cannot be exported as static HTML because route handlers proxy authenticated requests, verify Ed25519 certificates, read server-only configuration, and manage HttpOnly cookies.

## Environment split

| Variable | Phase | Exposure |
| --- | --- | --- |
| `NEXT_PUBLIC_TURNSTILE_SITE_KEY` | container build/public | Embedded for the browser; required and registered for the production Web hostname |
| `NEXT_PUBLIC_PINCHANA_WEB_COMMIT` | container build/public | Optional full Web Git revision shown and linked from Settings; defaults to `development` when no valid SHA is supplied |
| `PINCHANA_API_URL` | container runtime/server | Default API origin; required and never public |
| `PINCHANA_INSTANCE_PUBLIC_KEY` | container runtime/server | Optional Ed25519 public verification PEM; preserve newlines or use escaped `\n` |
| `PINCHANA_WEB_IMAGE` | Compose build | Local image tag, default `pinchana-web:latest` |
| `WEB_BIND_ADDRESS` | Compose host | Host interface, default `127.0.0.1` |
| `WEB_HOST_PORT` | Compose host | Reverse-proxy target port, default `3000` |

Configure the selected API separately with the matching Turnstile secret, expected Web hostname `pinchana.example.com`, action `turnstile-spin-v1`, and signed-session secret.

## Start the service

<Steps>
1. Clone [Pinchana Web](https://github.com/Pinchana/pinchana-web) on the Docker host.
2. Create a mode-0600 `.env` beside `docker-compose.yml` with the required variables. Compose reads it for interpolation; it is not copied into the image.
3. Run `docker compose --env-file .env config --quiet`. Inspect the full effective configuration only in a private terminal because it contains deployment values.
4. Build with the exact revision, then start the immutable image.
5. Wait for `docker compose ps` to report the service healthy, then proxy HTTPS to `127.0.0.1:3000`.
</Steps>

```ini
PINCHANA_API_URL=https://api.pinchana.cc
NEXT_PUBLIC_TURNSTILE_SITE_KEY=REPLACE_WITH_PUBLIC_SITE_KEY
NEXT_PUBLIC_PINCHANA_WEB_COMMIT=REPLACE_WITH_FULL_WEB_GIT_COMMIT
PINCHANA_INSTANCE_PUBLIC_KEY="-----BEGIN PUBLIC KEY-----\nREPLACE_WITH_PUBLIC_KEY\n-----END PUBLIC KEY-----"
WEB_BIND_ADDRESS=127.0.0.1
WEB_HOST_PORT=3000
```

The Docker build context excludes `.git`, so it cannot discover a revision by itself. Compose also cannot execute `git rev-parse` inside `.env`. Paste the full commit into `.env`, or inject it for the build:

```sh
NEXT_PUBLIC_PINCHANA_WEB_COMMIT="$(git rev-parse HEAD)" docker compose build web
docker compose --env-file .env up --detach web
docker compose --env-file .env logs --tail=100 web
```

The Dockerfile removes any stale `next-env.d.ts` path before `next build`, avoiding a host-created directory from breaking TypeScript generation. The final image contains the compiled application, production dependencies, public assets, and local FFmpeg worker/core; it does not bind-mount the source tree or install packages during container startup.

## Reverse proxy and updates

Put Caddy, Nginx, Traefik, or another HTTPS reverse proxy in front of the loopback port. Preserve streaming, disable response buffering for App Router streams, and forward byte-range requests used by the media proxy.

```sh
git pull --ff-only
NEXT_PUBLIC_PINCHANA_WEB_COMMIT="$(git rev-parse HEAD)" \
  docker compose --env-file .env up --detach --build web
docker compose --env-file .env logs --tail=100 web
```

The site key and revision are frozen into the build, so changing either requires rebuilding. Roll back by checking out the previous revision and rebuilding that image. Preserve the `.env`; there are no application data volumes.

## Verification checklist

- Page load checks an existing session or renders Turnstile.
- Browser network requests never reveal the API origin, signed session, or instance certificate cookie.
- URL processing, custom image/video/audio preview, byte-range seeking, Media and Audio-only saves, ZIP creation, and FFmpeg loading work.
- A YouTube job replaces the URL bar with progress, completes through the browser download manager, and supports pause/resume behavior provided by the browser and server range requests.
- Cookies are `HttpOnly`, `SameSite=Strict`, and `Secure` over production HTTPS.
- Custom origin selection accepts an authorized certificate and rejects mismatched/expired envelopes.

<Aside type="caution">
Do not add a machine `PINCHANA_API_KEYS` secret to Pinchana Web. Its browser flow is intentionally isolated and uses the API's signed `/web/...` session routes.
</Aside>
