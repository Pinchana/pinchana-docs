Machine clients should use `POST https://api.pinchana.cc/v1/scrape`. The request contains one complete HTTP(S) URL and requires a named `X-API-Key`. Machine credentials must remain in server-side configuration.

<Tabs syncKey="language">
<TabItem label="cURL">

```sh
export PINCHANA_API_KEY='REPLACE_WITH_API_KEY'

curl --fail-with-body --silent --show-error \
  --request POST \
  --header 'Content-Type: application/json' \
  --header "X-API-Key: ${PINCHANA_API_KEY}" \
  --data '{"url":"https://www.instagram.com/p/SHORTCODE/"}' \
  'https://api.pinchana.cc/v1/scrape'
```

</TabItem>
<TabItem label="JavaScript">

```js
const response = await fetch('https://api.pinchana.cc/v1/scrape', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'X-API-Key': process.env.PINCHANA_API_KEY,
  },
  body: JSON.stringify({url: 'https://www.instagram.com/p/SHORTCODE/'}),
  signal: AbortSignal.timeout(180_000),
});

const payload = await response.json();
if (!response.ok) {
  throw new Error(`${payload.error?.code ?? response.status}: ${payload.error?.message ?? 'Request failed'}`);
}

for (const asset of payload.data.media) {
  console.log(asset.index, asset.role, asset.type, asset.url);
}
```

<TabItem label="Python">

```py
import os

import httpx

with httpx.Client(timeout=180) as client:
    response = client.post(
        "https://api.pinchana.cc/v1/scrape",
        headers={"X-API-Key": os.environ["PINCHANA_API_KEY"]},
        json={"url": "https://www.instagram.com/p/SHORTCODE/"},
    )
    response.raise_for_status()
    payload = response.json()

for asset in payload["data"]["media"]:
    print(asset["index"], asset["role"], asset["type"], asset["url"])
```

<Steps>
1. Store the key in a server-side secret named for the client, such as `PINCHANA_API_KEY`.
2. Submit a supported post, track, or playlist URL. A 180-second client timeout permits the gateway's 120-second forwarding window plus network overhead.
3. Read assets from `data.media` in ascending `index` order. Select by `role` and `type` rather than by a platform-specific field.
4. Resolve relative media paths against the API origin and send the same `X-API-Key` for every media request.
</Steps>

The following command downloads the first known media path and verifies byte-range support:

```sh
curl --fail-with-body --silent --show-error --location \
  --header "X-API-Key: ${PINCHANA_API_KEY}" \
  --header 'Range: bytes=0-1023' \
  --output media.part \
  'https://api.pinchana.cc/media/instagram/SHORTCODE/video.mp4'
test -s media.part
```

<Aside type="caution">
Media URLs are authenticated, temporary cache handles. They are not permanent public content delivery network URLs. Submit the source URL again after cache eviction.
</Aside>

The unversioned `/scrape` route remains available for existing integrations. New clients must use API v1 so that success and failure responses have stable envelopes.
