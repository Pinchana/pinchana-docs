## API v1 error codes

Versioned scrape routes return `{"error":{"code":"...","message":"...","details":null}}`. Clients should branch on the stable code rather than matching the message.

| Code | Meaning | Client action |
| --- | --- | --- |
| `unsupported_url` | No configured module matches the URL | Check the complete URL and the operator's route patterns |
| `invalid_url` | The URL is malformed or rejected by the selected module | Ask for a complete supported post URL |
| `unauthorized` | The API key or Web session is absent or invalid | Restore the appropriate credential; do not retry anonymously |
| `forbidden` | The authenticated caller is not permitted | Stop and inspect the relevant authentication boundary |
| `authentication_required` | The upstream post requires an account or audience confirmation | Explain that anonymous extraction is unavailable |
| `restricted_media` | The platform confirms that the media is restricted | Do not perform repeated retries or VPN rotations |
| `not_found` | The source post or requested resource no longer exists | Ask for a valid existing URL |
| `rate_limited` | The upstream platform is limiting requests | Apply bounded backoff; do not create an unbounded retry loop |
| `extraction_failed` | The selected scraper could not extract the current upstream response | Record diagnostics and retry only when operational policy allows |
| `invalid_upstream_response` | A module returned data that the gateway could not normalize | Inspect gateway and module logs |
| `service_unavailable` | A required module or service is unavailable | Check aggregate health before retrying |
| `internal_error` | The scraper failed unexpectedly | Record the request identifier and inspect server logs |

Instagram and TikTok use at most one VPN rotation and one retry for confirmed blocking signals. Login-gated, removed, and ordinary anonymous misses do not receive repeated rotations.

## HTTP status codes

| Status | Typical cause |
| ---: | --- |
| 200 | Scrape, health, session, admin, or file request succeeded |
| 206 | A normal or DLP media endpoint satisfied a byte-range request |
| 400 | No module matches the URL; invalid custom-instance or Web BFF input |
| 401 | Missing/invalid machine key or browser bearer session |
| 403 | Turnstile verification failed |
| 404 | Cached file, module name, or safe path not found |
| 409 | DLP replay, invalid state transition, or file requested before ready |
| 410 | DLP worker key or job expired |
| 413 | DLP request body or cookie envelope exceeded its configured limit |
| 416 | Requested media byte range is not satisfiable |
| 422 | FastAPI/Pydantic request validation failure |
| 429 | A module or upstream platform is rate-limiting work |
| 500 | Unexpected VPN rotation or module failure |
| 501 | Dynamic container mode is disabled for a start/stop request |
| 502 | The Web BFF received an invalid verification response |
| 503 | Module, verification service, gateway client, VPN, auth configuration, identity, or aggregate health unavailable |

Legacy and non-versioned FastAPI errors may use `{"detail": ...}`. The official Web back end for front end maps failures to controlled `{"error":"..."}` messages and hides most server-side detail for 5xx errors.

## Cache lifecycle

Each module downloads media into `CACHE_PATH/{post_id}/` and writes `metadata.json` only after a successful result. Concurrent requests for the same key share one in-flight task. Partial directories without metadata are removed before retry. `CACHE_MAX_SIZE_GB` controls a shared size ceiling; before writing, the oldest post directory by file modification time is removed until enough space exists.

The Compose volume `scraper-cache` is shared by the gateway and modules, so a returned relative URL resolves to the gateway's local file. Eviction, operator cleanup, volume replacement, or a missing/incomplete download turns an old URL into `404`.

## Delivery and range requests

Machine clients must include `X-API-Key` on every media fetch. Browser clients use `/web/media/...` with a bearer session; the Next.js route exposes it as `/api/media/...`, forwards the incoming `Range` header, streams the body, and copies `Accept-Ranges`, content range/length/type/disposition, ETag, and last-modified headers. Its response is `Cache-Control: private, no-store`.

Do not publish authenticated cache URLs, assume they are permanent, or strip path components. Re-run `/v1/scrape` when media has been evicted.

DLP files are separate temporary artifacts rather than members of `scraper-cache`. They remain owner-bound to one signed browser session, are served with `no-store`, accept range requests while ready, and disappear when the job expires. Re-submit the source URL to create a new job; a normal `/scrape` cannot restore a DLP file.
